## 1.1 RISC-V基础SoC搭建

首先认识 Gem5 的基本构建流程。Gem5 环境配置与安装过程请参见其他文章，假设现在已经安装好 Gem5，现在开始我们的第一个 RISC-V CPU。

`\gem5\src\arch\riscv` 中包含 Gem5 关于 RISC-V 的源码，用户可以通过创建 Python 脚本的方式便捷调用 Gem5 的函数，用于搭建 RISC-V 的 CPU，首次使用需要编译 Gem5 的 RISC-V 源码。

在运行 Gem5 时，执行流程通常如下：

**Python 配置阶段**：解析你写的脚本，创建各种 `SimObject`（模拟对象）的实例，并将它们连接起来，如 CPU 连总线，总线连内存。

**C++ 实例化阶段**：当你调用 `m5.instantiate()` 时，Gem5 会根据 Python 中定义的拓扑结构，在 C++ 空间创建对应的真实模拟对象。

**事件驱动模拟阶段**：调用 `m5.simulate()`，C++ 引擎开始处理时序事件（Ticks），模拟硬件运行。

假设我们在 `\gem5\myriscv` 中创建一个脚本，命名为 `riscv_scalar.py`，接下来开始详细的代码流程。

### 1.1.1 导入库

```python
import os

import m5
from m5.objects import *
```

`m5` 库是 Gem5 的官方 Python 库，包含了所需的各类函数及类型。

### 1.1.2 创建一个系统对象

这段代码定义了模拟系统的**时序基础**和**层级结构**。

```python
system = System()
system.clk_domain = SrcClockDomain()
system.clk_domain.clock = "1GHz"
system.clk_domain.voltage_domain = VoltageDomain()
```

**(1) `system = System()`**

- **功能**：实例化一个 `System` 对象。
- **解析**：`System` 是 Gem5 中最顶层的容器对象。它包含了模拟系统中的所有硬件组件，如 CPU、内存控制器、总线等。可以把它想象成电脑的**主板**，后续所有的硬件都要挂载在这个 `system` 变量下。它还负责管理全局信息，比如内存范围、内核镜像路径等。

**(2) `system.clk_domain = SrcClockDomain()`**

- **功能**：为系统创建一个**源时钟域（Source Clock Domain）**。
- **解析**：在硬件设计中，不同的组件可能运行在不同的频率下。Gem5 通过“时钟域”来管理这些频率。`SrcClockDomain` 是一个产生时钟信号的源头。将其赋值给 `system.clk_domain` 意味着这个时钟域将作为该系统下所有组件的**默认时钟源**。

**(3) `system.clk_domain.clock = '1GHz'`**

- **功能**：设置该时钟域的运行频率为 **1GHz**。
- **解析**：Gem5 的模拟是基于 **Tick（滴答）** 的。1GHz 意味着每 1 纳秒产生一个 Tick。Gem5 内部会将频率转换为周期。如果后续没有为特定的 CPU 单独设置时钟，该 CPU 就会按照这个 1GHz 的频率运行。

**(4) `system.clk_domain.voltage_domain = VoltageDomain()`**

- **功能**：为该时钟域关联一个**电压域（Voltage Domain）**。
- **解析**：
  - 即使只关心性能模拟，Gem5 也要求关联电压域。这是因为 Gem5 集成了功耗建模，如 McPAT 或内部功耗模型。
  - **作用**：频率和电压共同决定了芯片的功耗。默认的 `VoltageDomain()` 通常设为 1.0V。如果在复杂的模拟中涉及到 DVFS，电压域的配置就变得至关重要。

### 1.1.3 创建存储器对象

这两行代码负责配置 Gem5 模拟系统中**最核心的内存子系统属性**。

```python
system.mem_mode = "timing"
system.mem_ranges = [AddrRange("512MB")]
```

**(1) `system.mem_mode = "timing"`**

- **含义**：将系统内存模式设置为“时序（Timing）”模式。
- **解析**：Gem5 在模拟内存时，主要有两种常用的模式：**Atomic（原子模式）** 和 **Timing（时序模式）**。
  - **Timing（时序模式）**：这是性能评估最常用的模式。在这个模式下，当 CPU 发起一个内存读写请求时，该请求会被打包成一个数据包。这个数据包需要“物理地”穿过缓存、排队等候总线、到达内存控制器，最后访问 DRAM。**它会精确模拟这一路上产生的每一纳秒的延迟、带宽限制以及资源冲突。**
  - **Atomic（原子模式）**：用于快速启动或纯功能验证。在这个模式下，内存访问是“瞬间完成”的。CPU 发出请求后，立刻就能拿到数据，不会模拟中间总线和控制器的延迟。它跑得极快，但无法用于评估系统性能。

**(2) `system.mem_ranges = [AddrRange("512MB")]`**

- **含义**：为系统分配一段大小为 512MB 的物理内存地址空间。
- **解析**：`AddrRange("512MB")` 是 Gem5 提供的一个辅助函数，用于定义一段物理地址。如果只传入大小，Gem5 会默认这段内存从物理地址 `0x00000000` 开始。
  - **为什么用列表包裹？**：因为真实的计算机系统物理内存往往不是一整块连续的。在复杂系统结构中，通常会有“内存空洞（Memory Holes）”被保留给 MMIO，比如显卡、网卡等设备寄存器。因此，Gem5 允许传入一个包含多个 `AddrRange` 的列表。

### 1.1.4 创建CPU并指定指令集架构

这两行代码是整个 Gem5 模拟脚本的**核心配置**。它们负责实例化一个具体的 CPU 微架构，并为其注入支持 RISC-V 的指令集规范。

Gem5 的 RISC-V 解码器默认就已经支持 **IMAFDC** 这些标准扩展。如果我们想要单纯模拟 RV32I 指令集，那么只能修改源码去掉其他指令支持使其报指令异常，或者在编译代码的时候通过 RISC-V 编译器指定。后者只是“在 RV32GC 的硬件上跑 RV32I 指令”，并不是真正的 RV32I。

```python
system.cpu = RiscvTimingSimpleCPU()
system.cpu.isa = [
    RiscvISA(
        riscv_type="RV32",
    )
]
```

**(1) `system.cpu = RiscvTimingSimpleCPU()`**

**功能**：在系统中创建一个名为 `cpu` 的组件，类型为 `RiscvTimingSimpleCPU`。

**解析**：这个名字包含了三个关键信息：

1. **Riscv**：表明这是一个专为 RISC-V 架构定制的 CPU 壳子。
2. **Simple**：指的是**简单微架构**。这是一个单发射、顺序执行的 CPU。它没有现代 CPU 中复杂的乱序执行引擎、深度流水线或高级分支预测器。它一次只执行一条指令。
3. **Timing**：指的是**时序内存模型**。它必须与前文提到的 `system.mem_mode = "timing"` 配合使用。当这条简单的 CPU 遇到 `Load/Store` 指令时，它会真正地“停顿”，等待模拟的内存系统把数据取回来后，才继续执行下一条指令。

**(2) `system.cpu.isa = ...`**

**功能**：创建一个 `RiscvISA` 对象，把这个 ISA 赋予给刚才创建的 CPU。之所以用列表 `[]`，是因为 Gem5 支持多线程/多核场景下为每个硬件线程分配 ISA 上下文。

**解析**：`riscv_type` 决定了 CPU 基础整数寄存器是 32 位还是 64 位宽。

### 1.1.5 创建内存总线及配置CPU侧端口

这四行代码是 Gem5 硬件拓扑构建中的**接线阶段**。它们负责建立系统的主干道，并将 CPU 的各个接口接到这条总线上。在 Gem5 中，组件之间的通信完全依赖于**端口（Ports）**的连接。

```python
system.membus = SystemXBar()
system.cpu.icache_port = system.membus.cpu_side_ports
system.cpu.dcache_port = system.membus.cpu_side_ports
system.cpu.createInterruptController()
```

**(1) `system.membus = SystemXBar()`**

- **功能**：创建一个名为 `membus` 的 `SystemXBar` 对象。
- **解析**：**XBar（Crossbar Switch）**是一种高度并行的互联网络。把它想象成一个高速立交桥或网络路由器。当多个设备同时想访问内存时，XBar 负责仲裁、排队并路由这些请求。

**(2) `system.cpu.icache_port = system.membus.cpu_side_ports`**

**含义**：将 CPU 的指令取指端口连接到系统总线的 CPU 侧端口。

**解析**：

- 注意 CPU 端的端口叫 `icache_port`，但这**并不代表系统里有缓存**。它只是 CPU 用来抓取指令的“物理引脚”的名字。
- 在 Gem5 的端口配置中，`=` 被重载为“连接”。这就相当于拿起一根数据线，一头插在 CPU 的 `icache_port` 上，另一头插在总线提供给 CPU 的插槽上。

**(3) `system.cpu.dcache_port = system.membus.cpu_side_ports`**

**含义**：将 CPU 的数据读写端口也连接到系统总线的 CPU 侧端口。

**解析**：

- 现代 CPU 通常采用哈佛架构的对外接口，即“取指令”和“读写数据”在物理上是两条独立的通路。
- 这行代码将数据线也接到了同一个 `membus` 的插槽池里。这样，无论是指令抓取还是数据加载/存储，都会汇聚到这个 `SystemXBar` 进行统一调度。

**(4) `system.cpu.createInterruptController()`**

**含义**：为 CPU 创建并连接中断控制器。

**解析**：

- 即使运行的是不需要 Linux 内核的裸机或简化场景，CPU 在执行指令时依然可能遇到异常，比如非法指令或者系统调用。
- 调用这个辅助函数不仅仅是实例化一个中断控制器对象，它还会**自动在底层将中断控制器的端口连接到 CPU 和内存总线上**。如果不加这一行，模拟器往往会直接崩溃报错。

### 1.1.6 创建内存及配置存储器端口和地址

这五行代码完成了虚拟硬件构建的最后一步：**配置物理内存并将其接入系统总线**。

```python
system.mem_ctrl = MemCtrl()
system.mem_ctrl.dram = DDR3_1600_8x8()
system.mem_ctrl.dram.range = system.mem_ranges[0]
system.mem_ctrl.port = system.membus.mem_side_ports
system.system_port = system.membus.cpu_side_ports
```

**(1) `system.mem_ctrl = MemCtrl()`**

`MemCtrl()` 是内存控制器的通用模型。它负责管理内存请求的排队、调度以及刷新操作。

**`DDR3_1600_8x8()`** 定义了具体的内存特性：

- **DDR3**：内存标准。
- **1600**：数据传输速率。
- **8x8**：指内存芯片的组织结构。

Gem5 内置了多种模型，如 DDR3、DDR4、LPDDR、HBM 等。选择不同模型会直接影响程序的访存延迟和系统带宽。

**(2) `system.mem_ctrl.dram.range = system.mem_ranges[0]`**

**功能**：将之前定义的 `512MB` 地址范围分配给这块 DDR3 内存。

**解析**：这行代码告诉内存控制器，凡是 CPU 发出的、落在该物理地址范围内的请求，都由它来负责处理。

**(3) `system.mem_ctrl.port = system.membus.mem_side_ports`**

**功能**：将内存控制器的端口连接到总线的**内存侧端口**。

**解析**：

- 在前一部分中，我们将 CPU 连接到了总线的 `cpu_side_ports`。
- 现在，我们将内存连到了 `mem_side_ports`。
- 至此，数据链路闭环：**CPU <-> 总线 <-> 内存控制器**。

**(4) `system.system_port = system.membus.cpu_side_ports`**

**功能**：连接 Gem5 特有的“系统端口”到总线。

**解析**：

- `system_port` 不是物理硬件，而是 **Gem5 模拟器自身的“后门”**。
- 在模拟开始前，Gem5 需要将你的二进制文件加载进模拟内存里。
- `system_port` 允许模拟器在不通过 CPU 指令的情况下，直接通过总线向内存读写数据。如果没有这行代码，程序甚至无法被正确加载到虚拟内存中。

### 1.1.7 指定加载的程序和运行模式

```python
binary = os.path.abspath("./test/hello32")

system.workload = RiscvBareMetal()
system.workload.bootloader = binary

system.cpu.createThreads()

root = Root(full_system=True, system=system)

m5.instantiate()

exit_event = m5.simulate()
```

- **`binary = os.path.abspath("./test/hello32")`**：指定要运行的程序。
- **`system.workload = RiscvBareMetal()`**：使用裸机模式运行负载。
- **`system.workload.bootloader = binary`**：加载运行程序。
- **`system.cpu.createThreads()`**：为 CPU 创建硬件线程上下文。这包含程序计数器、通用寄存器等。没有这一步，CPU 就没有存放运行状态的地方。
- **`Root`**：创建模拟树的根节点。这里的 `full_system=True` 与 `RiscvBareMetal()` 结合使用，表示以完整系统树结构去承载当前这套裸机运行环境。
- **`m5.instantiate()`**：实例化。在执行这行代码之前，前面写的 `System()`、`SimpleCPU()`、`MemCtrl()` 都只是 Python 对象，用来收集参数。当这行代码执行时，Gem5 引擎会遍历整个 Python 对象树，并在底层的 C++ 空间中真正创建这些硬件模型。
- **`m5.simulate()`**：通电开机。调用这个函数后，Python 脚本会阻塞，Gem5 的 C++ 事件队列开始运行，模拟时间开始向前推进。
- **`exit_event`**：`simulate()` 函数会一直运行，直到遇到一个退出事件才会把控制权交还给 Python。退出事件可能是程序正常结束、程序出错，或者达到预设的最大 Tick 数。

## 1.2 RISC-V简易测试程序编写与测试

关于 RISC-V 的编译链配置略过，我们的目标是使用 C 语言编写一个最简单的 "hello, world!" 打印程序。在裸机运行中打印文字并不能直接写 `printf`，而是需要通过在总线上挂载串口，向串口地址写字符的形式来打印。除此之外，我们还需要一个启动文件 `Start.S` 和链接文件 `link.ld` 来指示编译器完成初始化和内存地址分布。

总结裸机运行 "print" 的流程为：

1. 在 Gem5 的 CPU 上通过总线挂载一个 UART 设备，约定 UART 地址。
2. 准备 `Start.S` 启动文件，主要用于设置堆栈地址，清空 `.bss` 等。
3. 准备 `link.ld` 链接文件，用于设置代码段、数据段地址。
4. 编写裸机程序代码，向 UART 地址写入字符。

### 1.2.1 为Gem5添加串口设备

按照以上流程，首先进行第一步，在 Gem5 中加入总线上的串口设备。这里模拟使用 HiFive 开发板的硬件，也可以自定义串口设备：

```python
system.platform = HiFive()
system.platform.terminal.outfile = "stdoutput"
system.platform.uart.device = system.platform.terminal
system.platform.setNumCores(1)
system.platform.attachPlic()
system.platform.pci_host.pio = system.membus.mem_side_ports
system.platform.attachOnChipIO(system.membus)
system.platform.attachOffChipIO(system.membus)
```

**(1) `system.platform = HiFive()`**

- **含义**：实例化一个 SiFive HiFive 匹配的平台模型。
- **解析**：HiFive 是 RISC-V 世界里最著名的开发板系列。当你指定了这个平台，Gem5 会自动在系统里预留出一套标准的 RISC-V 内存映射表。例如，它规定了 UART 必须在 `0x10000000`，CLINT 必须在 `0x02000000`。

**(2) 终端与串口重定向**

- **含义**：将虚拟硬件的串口输出重定向到宿主机的文件。
- **解析**：
  - `terminal` 是一个抽象层，用来模拟控制台。
  - `outfile = "stdoutput"` 表示模拟器运行产生的字符会写入一个叫 `stdoutput` 的文件。
  - `uart.device = ...` 则建立起 UART 和终端之间的连接。

**(3) `system.platform.setNumCores(1)`**

- **含义**：告诉平台只有 1 个 CPU 核心。
- **解析**：硬件平台特别是中断控制器，需要知道有多少个核心在监听中断。如果这里设置错了，中断信号可能永远传不到 CPU。

**(4) `system.platform.attachPlic()`**

- **含义**：安装 **PLIC（Platform-Level Interrupt Controller）**。
- **解析**：PLIC 是 RISC-V 标准的平台级中断控制器。它负责收集外设发出的中断信号，并有秩序地分发给 CPU。在裸机开发中，它是处理 I/O 的核心。

**(5) `system.platform.pci_host.pio = system.membus.mem_side_ports`**

- **含义**：将 PCI 控制器的寄存器接口连接到总线。
- **解析**：虽然 HiFive 基础版不一定用到复杂的 PCI 设备，但为了系统完整性，这里依然将 PCI 的配置空间映射到了内存总线上。

**(6) I/O 设备的自动化布线**

- **`attachOnChipIO`**：将片上外设，如 CLINT 计时器、PLIC 中断控制器，连接到系统总线。
- **`attachOffChipIO`**：将片外外设，如 UART 串口、磁盘控制器、实时时钟，连接到系统总线。

### 1.2.2 添加启动文件和链接脚本

添加 `Start.S` 作为启动文件，设置堆栈地址，清空 `.bss`：

```assembly
.section .text.start
.globl _start
_start:
    /* Initialize stack pointer from linker symbol. */
    la sp, __stack_top

    /* Zero .bss so C runtime assumptions hold. */
    la t0, __bss_start
    la t1, __bss_end
1:
    bgeu t0, t1, 2f
    sw zero, 0(t0)
    addi t0, t0, 4
    j 1b

2:
    call main

3:
    j 3b
```

添加 `link.ld` 作为链接脚本，设置代码段和数据段起始地址及大小：

```ld
OUTPUT_ARCH(riscv)
ENTRY(_start)

MEMORY
{
    /* Typical RISC-V memory map: RAM at 0x80000000, MMIO below. */
    RAM (rwx) : ORIGIN = 0x80000000, LENGTH = 0x08000000
}

PHDRS
{
    text PT_LOAD FLAGS(5); /* R-X */
    data PT_LOAD FLAGS(6); /* RW- */
}

SECTIONS
{
    . = ORIGIN(RAM);

    .text : ALIGN(4)
    {
        KEEP(*(.text.start))
        *(.text .text.*)
    } > RAM :text

    .rodata : ALIGN(4)
    {
        *(.rodata .rodata.*)
    } > RAM :text

    . = ALIGN(0x1000);

    .data : ALIGN(4)
    {
        __data_start = .;
        *(.data .data.*)
        *(.sdata .sdata.*)
        __data_end = .;
    } > RAM :data

    .bss (NOLOAD) : ALIGN(4)
    {
        __bss_start = .;
        *(.sbss .sbss.*)
        *(.bss .bss.*)
        *(COMMON)
        __bss_end = .;
    } > RAM :data

    . = ALIGN(16);
    __stack_top = ORIGIN(RAM) + LENGTH(RAM);

    /DISCARD/ :
    {
        *(.comment)
        *(.note*)
        *(.eh_frame*)
    }
}
```

### 1.2.3 写C代码驱动串口输出

现在需要使用 C 语言向串口地址发送字符完成输出，如下所示：

```c
#define UART_ADDR 0x10000000
volatile char* uart = (volatile char*)UART_ADDR;

void print_uart(const char* s) {
    while (*s) {
        *uart = *s;
        s++;
    }
}

int main() {
    print_uart("Hello, World!\n");
    while(1);
}
```
