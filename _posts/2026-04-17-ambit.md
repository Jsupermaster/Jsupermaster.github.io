> 论文：Ambit: In-Memory Accelerator for Bulk Bitwise Operations Using Commodity DRAM Technology (MICRO'17)

### 一、摘要

这篇论文关注的是大规模按位运算（bulk bitwise operations）在数据密集型应用中的加速问题。作者指出，数据库位图索引、BitWeaving、Web 搜索中的 BitFunnel、DNA 比对、图处理和加密等场景都大量依赖 AND、OR、NOT、XOR 等位操作，但现有 CPU、GPU 乃至 3D-stacked memory logic layer 的实现都受限于外部内存带宽，主要瓶颈不是计算本身，而是数据在 DRAM 与处理器之间的搬运。为了解决这一问题，论文提出 Ambit，一种直接在 commodity DRAM 阵列内部执行 bulk bitwise operation 的 in-DRAM accelerator。其核心做法是利用 triple-row activation 在 DRAM 内实现 AND/OR，利用 dual-contact cell 和 sense amplifier 的互补节点实现 NOT，再结合 designated rows、RowClone、保留地址映射和 split row decoder，把整个方案做成低成本、可直接挂接到现有内存总线上的实用设计。实验结果表明，Ambit 在七类 bulk bitwise operation 上平均带来约 32 倍性能提升和 35 倍能耗降低，并在 bitmap indices、BitWeaving 和 bitvector-based set operations 三类真实应用中分别获得约 6.0 倍、7.0 倍和 3.0 倍的端到端加速。

### 二、现有问题

**1. 大规模按位运算广泛存在，但传统执行方式受内存带宽严重限制**

- 数据库位图索引、BitWeaving、BitFunnel、DNA 序列比对、图处理、加密算法等场景都依赖对大位向量做批量位运算。
- 在现有系统中，数据必须先从 DRAM 传到 CPU、GPU 或其他处理单元后再做按位操作。
- 因为数据量很大，吞吐通常不是被 ALU 限制，而是被 memory channel bandwidth 限制，导致延迟高、能耗高、吞吐低。

**2. 即使把计算下沉到 3D-stacked memory 的 logic layer，仍然没有充分利用 DRAM 阵列内部带宽**

- 近内存计算或逻辑层 PIM 确实比传统 CPU/GPU 更接近数据，但仍然依赖接口化的数据传输路径。
- 论文认为，logic layer 仍无法直接利用 DRAM 芯片内部 row buffer 和 subarray 所提供的最大内部带宽。
- 因此，对于 bulk bitwise operation，这类方案仍然没有从根本上消除带宽瓶颈。

**3. 真正把位运算做到 DRAM 内部，会遇到可靠性、破坏性和实现成本问题**

- triple-row activation 虽然可用于实现多数决逻辑，但 bitline 偏移更小，可能影响 sense amplifier 的判决可靠性。
- 三行同时激活后，参与操作的三行都会被最终结果覆盖，因此会破坏源数据。
- DRAM cell 会泄漏电荷，如果参与操作的行状态不够“新鲜”，运算正确性可能受影响。
- 若要支持任意三行同时激活，需要更宽地址总线和更复杂 row decoder，硬件代价太高。

**4. 配套的数据搬运和系统集成本身也可能成为瓶颈**

- 若源数据搬运、结果回写和控制行初始化仍通过内存控制器完成，复制开销会显著削弱收益。
- 在系统层面，缓存一致性、地址映射、ECC 兼容性和数据 scrambling 都会影响方案能否真正落地。

### 三、解决方法

为了克服上述问题，作者提出了 Ambit，一种直接在 DRAM 芯片内部执行 bulk bitwise operations 的低成本加速器。其核心设计如下：

**1. 利用 triple-row activation 在 DRAM 中实现 bulk AND / OR**

- 现代 DRAM 中，多行共享同一组 sense amplifiers。
- 当三行同时激活时，sense amplifier 最终得到的是三个 cell 的按位多数决结果。
- 如果第三行预设为全 0，则多数决等价于前两行按位 AND。
- 如果第三行预设为全 1，则多数决等价于前两行按位 OR。

**2. 利用 dual-contact cell 在 DRAM 中实现 bulk NOT**

- sense amplifier 在稳定后，bitline 和 bitline bar 上天然保存互补值。
- 论文引入 dual-contact cell（DCC），使一个 cell 可以连接 bitline 或 bitline bar。
- 这样就可以把某一行数据的反相结果直接写入 DCC，再复制到目标行，从而完成 bulk NOT。

**3. 使用 designated rows 和 RowClone，解决源数据覆盖与复制开销问题**

- 每个 subarray 预留若干 designated rows，如 `T0-T3`，用于暂存源操作数和中间结果。
- 预留 `C0/C1` 作为全 0 / 全 1 控制行，用于控制 AND / OR 的逻辑含义。
- 预留 DCC 行用于执行 NOT。
- 在真正进行位运算之前，先通过 RowClone 把源数据复制到 designated rows，再执行运算，避免覆盖原始数据。
- 借助 RowClone-FPM，复制和初始化可以在 DRAM 内部高效完成。

**4. 通过保留地址映射和 split row decoder，把方案做成低成本硬件实现**

- Ambit 不支持任意三行组合，而是只允许在预定义的一小组专用行上做 triple-row activation。
- 通过保留 row addresses，内存控制器只需发出普通 `ACTIVATE` 命令，芯片内部就能触发预定义的三行同时激活。
- row decoder 被拆成普通数据行和 bitwise 专用行两部分，降低实现复杂度，并缩短 AAP 执行延迟。

**5. 沿用标准 DRAM 接口，直接挂接到系统 memory bus**

- Ambit 不修改 DRAM command/address interface。
- 因此它可以像普通 DRAM 一样接到系统 memory bus，而不是作为独立 PCIe 加速器存在。
- 应用可通过处理器指令直接触发 Ambit 操作，减少主机和加速器内存之间的额外数据搬运。

> ### DRAM基础知识与基本结构
>
> ![image-20260412183444080]({{ '/assets/images/ambit/image-20260412183444080.png' | relative_url }})
>
> DRAM中包含多个bank，每个bank中包含多个cell array。在一次读取中会激活一整行cell并打开这一整行的晶体管，从而让这一行的电容微弱放电，通过感测放大器来检测位线的微弱变化并将其彻底拉高或拉低，从而读出一行数据，这个过程同时为这一行电容重新充电。最后根据列地址读出所需的一位数据。
>
> 正常的DRAM只会激活1行数据，而论文中提出的Ambit则会同时激活3行数据，这样的话感测放大器得到的是这三行数据的多数决。
>
> ![image-20260412184904004]({{ '/assets/images/ambit/image-20260412184904004.png' | relative_url }})
>
> DRAM 的 sense amplifier 本质上就是一个由两个交叉耦合反相器组成的双稳态锁存器。它一旦被使能，就只愿意停在两种稳定状态之一：
>
>   - bitline = 1, bitline bar = 0
>   - bitline = 0, bitline bar = 1
>
> 因此利用这一特性天然适合NOT。但常规的DRAM只会挂载其中一个，且无法更改，因此虽然bitline和bitline bar天然反相，但常规DRAM无法利用这一特性，本文的DCC能够同时利用这一对差分节点，从中选一组利用。
>
> ### 当三行同时激活时，为什么sense amplifier 最终得到的是三个 cell 的按位多数决结果
>
> 从实现原理来看，激活一行数据之前，位线被预充电到逻辑电平0和逻辑电平1的中间值，晶体管开启后，电容器对位线充电/放电，导致位线电压略高于VDD/2或略低于VDD/2，感测放大器只负责捕捉这一微弱变化，并据此将其设置为VDD或0。当同时激活三行数据时，有3个电容器同时影响同一根位线，做了电荷共享，最终结果就是3个电容器共同作用导致位线略高于VDD/2或略低于VDD/2，感测放大器依然捕捉这个变化，这个过程就相当于多数决。
>
> 1. 三个 cell 同时连到同一条 bitline 时，先发生的是电荷共享
>   2. sense amplifier 只负责把“高于还是低于 VDD/2”这个微小偏差放大成完整的 1/0
>
>   所以它最后得到的不是“三个值里某一个”，而是“三个值合起来后把 bitline 推到中点上方还是下方”，这恰好就是多数决。
>
>   更具体地说，普通 DRAM 在预充电后，bitline 先停在 VDD/2。如果只激活一行：
>
>   - cell 存 1，就把 bitline 往 VDD 方向拉一点
>   - cell 存 0，就把 bitline 往 0 方向拉一点
>
>   随后 sense amplifier 这个交叉耦合反相器锁存器会把这个很小的偏差放大：
>
>   - 稍高于 VDD/2，放大成完整 1
>   - 稍低于 VDD/2，放大成完整 0
>
>   Ambit 的 triple-row activation 只是把“一个 cell 给 bitline 注入电荷”变成“三个 cell 一起给 bitline 注入电荷”。
>
> 通过把3个cell中的1个固定设置成0或1，就可以完成AND或OR。当一个cell固定是1，那么只要剩下两个cell中至少有一个1，那结果就是1；如果两个都是0则结果是0，完成了OR的语义。当一个cell固定是0，只有剩下两个cell都是1结果才可能是1，完成了AND的语义。

### 四、技术细节

#### 1. Ambit-AND-OR：利用 triple-row activation 实现 AND / OR

![image-20260412184953290]({{ '/assets/images/ambit/image-20260412184953290.png' | relative_url }})

在一个 subarray 中，许多行共用一排 sense amplifiers。Ambit 让三行同时激活，形成 triple-row activation（TRA）。此时：

- 如果三行中至少两行该位为 `1`，charge sharing 后 bitline 电压会高于 `VDD/2`，sense amplifier 最终输出 `1`；
- 如果三行中至少两行该位为 `0`，bitline 电压会低于 `VDD/2`，sense amplifier 最终输出 `0`。

因此，TRA 本质上实现的是逐位 majority function。作者进一步利用一个控制行把 majority 变成目标逻辑：

- `A, B, 0` 的多数决等价于 `A AND B`
- `A, B, 1` 的多数决等价于 `A OR B`

这就是 Ambit-AND-OR 的核心。

#### 2. 让 TRA 真正可用：designated rows、控制行和刷新语义

直接拿真实数据所在行做 TRA 有三个问题：会覆盖源数据、会受 cell leakage 影响、还要求控制器一次指定三个任意地址。论文给出的解决方式是为每个 subarray 预留一小组 designated rows。

- `T0-T3`：作为临时操作数与中间结果缓冲区
- `C0`：预初始化为全 `0`
- `C1`：预初始化为全 `1`

这样，计算 `Dk = Di AND Dj` 时，流程不是直接对 `Di` 和 `Dj` 做 TRA，而是：

1. 把 `Di` 复制到 `T0`
2. 把 `Dj` 复制到 `T1`
3. 把 `C0` 复制到 `T2`
4. 对 `T0/T1/T2` 做 TRA
5. 把结果从 `T0` 复制到 `Dk`

这种做法有三个好处：

- 源数据不会被覆盖，因为真正参加 TRA 的是 designated rows。
- 复制本身等价于刚访问过这些行，因此 `T0/T1/T2` 在执行 TRA 时几乎处于“刚刷新”的状态，降低了 leakage 带来的风险。
- 由于 designated rows 是预定义的，控制器只需要发一个保留地址，就能触发固定的三行同时激活，不必支持“任意三行地址同时输入”。

> ### Designated rows设计的讨论
>
> 两个疑问：1. 给每个subarray预留5行左右作为designated rows，使得存储容量损失了一部分，且是否会造成软件调度的额外负担（比如要控制不可向这5行写入非指定数据）；2. 每次计算都要做几次复制，是否带来额外开销，这个复制是DDR芯片自己控制还是由主机控制？
>
> #### **1. 预留行问题：容量损失与软件负担**
>
> **1.1 需要预留哪些行？**
>
> Ambit 并非只预留“5行”，实际需要预留以下几类特殊行：
>
> - **T0-T3**：4行临时行，用于暂存运算的中间数据。
> - **C0/C1**：2行控制行，分别固定存储全0和全1模式。
> - **DCC 相关行**：用于实现 NOT 操作。
> - **总计**：每个子阵列（subarray）额外开销约 **8 行**。
>
> **1.2 容量损失大吗？**
>
> - 相对于 DRAM 巨大的总行数，这8行的容量/面积损失 **不到 1%**。
> - 论文认为此开销很小，是可以接受的折衷。
>
> **1.3 软件负担在哪里？**
>
> - **对普通软件透明**：这些预留行通过硬件保留地址和内部映射实现，**对操作系统和普通应用程序不可见**，不会被分配为普通物理页。软件无需“刻意避开”它们。
> - **真正的软件负担是数据布局优化**：为了获得最佳性能，软件（如驱动或智能分配器）应尽量将未来需要一起进行位运算的位向量（bitvector）段，**放置在同一子阵列内**。这样，后续的数据复制才能使用最快的内部复制机制（RowClone-FPM）。
> - **核心结论**： **正确性负担在硬件**：“禁止写入保留行”由内存控制器和地址映射机制负责，不落在应用软件上。 **性能优化负担在软件**：软件需要为性能考虑数据摆放（placement），而非为正确性规避保留行。
>
> #### **2. 复制开销问题：为何复制及如何控制**
>
> **2.1 为什么必须复制数据？**
>
> - 核心运算操作 **TRA** 是破坏性的。三行同时激活后，所有参与行都会被运算结果覆盖。
> - 因此，**不能直接对原始数据行进行运算**，必须先将源数据复制到临时行（T0, T1等），在临时行上完成运算。
>
> **2.2 一次运算的步骤（以 AND 为例）**
>
> 一次 `AND(Di, Dj -> Dk)`操作并非一条命令完成，典型流程如下：
>
> 1. `Di -> T0`(复制Di到临时行T0)
> 2. `Dj -> T1`(复制Dj到临时行T1)
> 3. `C0 -> T2`(复制全0行C0到T2，作为初始结果行)
> 4. `TRA(T0, T1, T2)`并写回 `Dk`(执行运算，结果存回目标行Dk)
>
> **2.3 复制的开销与实现**
>
> - **有开销，但相对较小**：复制步骤带来了固定开销，但相比将整行数据搬运到CPU/GPU再运算的传统方式，开销小得多。
> - **复制在DRAM内部完成**：数据复制不通过外部DDR通道。 同一子阵列内的复制使用 **RowClone-FPM** 机制。 本质是背靠背（back-to-back）的 ACTIVATE 命令，数据通过行缓冲（row buffer）和感应放大器（sense amplifier）在芯片内部移动。
>
> **2.4 控制关系：谁下命令？谁执行？**
>
> - **命令发起方**：**主机内存控制器**。它发送标准的DRAM命令（如ACTIVATE, PRECHARGE）序列。
> - **执行方**：**DRAM芯片内部电路**。芯片在识别到对特定保留地址的访问时，将其解释为特殊的内部操作（如RowClone、TRA、DCC写入）。
>
> **2.5 技术优势与局限判断**
>
> - **优势场景**：对于**大块、行对齐、较长的位向量**的批量位运算，Ambit 的内部复制和计算成本远低于传统方案的外部数据搬运成本。
> - **局限场景**：如果数据**很小、很零碎、或分散在多个子阵列**，需要跨子阵列复制，则 Ambit 的性能优势会明显下降。

#### 4. RowClone：把复制和初始化开销压低到 DRAM 内部

Ambit 的 AND / OR 不是一步完成的，而是“复制源数据 + 准备控制行 + 执行 TRA + 回写结果”的组合。因此，如果复制还走传统内存通道，收益很容易被搬运开销吃掉。

为此，论文直接复用了 RowClone：

- `RowClone-FPM`：在同一 subarray 内通过两个 back-to-back `ACTIVATE` 完成快速行复制，延迟约 80ns。
- `RowClone-PSM`：跨 bank 通过内部总线复制，虽然比经由内存控制器更快，但慢于 FPM。

Ambit 的设计目标是尽量让 bulk bitwise 涉及的数据位于同一 subarray，这样大多数复制都可以走 `RowClone-FPM`。为此论文还提出：

- 在每个 subarray 中都放置 designated rows 和控制行；
- 由驱动感知底层地址映射，把未来可能参与位运算的 bitvector 映射到同一 subarray；
- 对于很长的 bitvector，可以跨多个 subarray 交错存放，但要求不同 bitvector 的对应分段落在同一 subarray 中。

#### 5. Ambit-NOT：利用 dual-contact cell 读取感放器的反相信号

![image-20260412214649479]({{ '/assets/images/ambit/image-20260412214649479.png' | relative_url }})

AND / OR 可以通过 majority 完成，但 NOT 不能直接靠 TRA 得到。Ambit 的做法是利用 sense amplifier 天然同时产生原值与反相值这一事实。

论文引入 dual-contact cell（DCC），它是一个 `2T-1C` cell，有两条独立接入路径：

- `d-wordline`：把 DCC 接到 bitline
- `n-wordline`：把 DCC 接到 bitline bar

其工作流程如下：

1. 激活源行 `A`
2. 此时 bitline 表示 `A`，bitline bar 表示 `NOT A`
3. 激活 DCC 的 `n-wordline`
4. 将 bitline bar 上的反相值写入 DCC
5. `PRECHARGE`
6. 再把 DCC 中保存的值复制到目标行 `R`

因此，Ambit-NOT 本质上是“先从感放器的反相侧取数，再把这个值拷到目标行”。论文指出，每个 DCC row 的面积大约等于两行普通 DRAM row 的代价。

![image-20260412214822431]({{ '/assets/images/ambit/image-20260412214822431.png' | relative_url }})

#### 6. Row Address Grouping：把复杂内部行为藏在保留地址后面

为了让控制器仍然只发标准 DRAM 命令，Ambit 对每个 subarray 的 row address space 做了重新分组：

- `B-group`：bitwise group，对应 `T0-T3` 和两组 DCC
- `C-group`：control group，对应 `C0` 和 `C1`
- `D-group`：data group，对应正常应用数据

在论文给出的一个 1024-row subarray 设计中：

- `D-group` 仍然保留 1006 行给应用数据；
- `B-group` 占用 16 个保留地址 `B0-B15`；
- `C-group` 占用两个地址 `C0/C1`。

其中最关键的地址映射包括：

- `B0-B3`：分别激活 `T0-T3`
- `B4/B6`：激活两组 DCC 的 `d-wordline`
- `B5/B7`：激活两组 DCC 的 `n-wordline`
- `B12-B15`：触发三行同时激活，例如 `B12 -> T0,T1,T2`
- `B8-B11`：触发双行同时激活，用于减少复合运算中的复制步骤

这样，内存控制器看到的仍然只是普通 `ACTIVATE B12`，而 DRAM 芯片内部实际执行的是“三条特定 wordline 同时拉起”。

#### 7. AAP / AP 原语：把所有位运算编排成命令序列

论文把 Ambit 的执行基本单元抽象成：

- `AAP(addr1, addr2)` = `ACTIVATE addr1; ACTIVATE addr2; PRECHARGE`
- `AP(addr)` = `ACTIVATE addr; PRECHARGE`

其中，`AAP` 的逻辑含义是：先让 `addr1` 对应的行或内部逻辑产生结果，再在仍处于激活过程中的阵列上，用第二次 `ACTIVATE` 把结果写到 `addr2` 对应的行中。

几个典型操作的命令序列如下：

**1. AND**

- `AAP(Di, B0)`：`T0 = Di`
- `AAP(Dj, B1)`：`T1 = Dj`
- `AAP(C0, B2)`：`T2 = 0`
- `AAP(B12, Dk)`：对 `T0/T1/T2` 做 TRA，并把结果写到 `Dk`

**2. NAND**

- 前三步与 `AND` 相同
- `AAP(B12, B5)`：把 `T0 AND T1` 的反相值写到 DCC
- `AAP(B4, Dk)`：把 DCC 中的值拷回 `Dk`

**3. XOR**

论文把 `XOR` 分解为：

- `(Di AND NOT Dj) OR (NOT Di AND Dj)`

因此需要：

- 同时保存 `Di` 和 `NOT Di`
- 同时保存 `Dj` 和 `NOT Dj`
- 使用额外 designated rows `T2/T3` 和第二组 DCC 存中间结果

这也是为什么论文在最小三行之外，实际设计里进一步扩展到 `T0-T3` 和两组 DCC。

#### 8. Split Row Decoder：为什么能把 AAP 从 80ns 降到 49ns

若朴素串行执行一个 `AAP`，延迟约为：

- `2 * tRAS + tRP`

对 DDR3-1600，大致约为 80ns。作者提出 split row decoder 的依据有两点：

- `AAP` 的第二次 `ACTIVATE` 发生在 bank 已经处于激活态时，不再需要完整的 sense amplification。
- 在大部分 AAP 中，两个地址里恰好只有一个属于 `B-group`，另一个属于 `C/D-group`。

于是可以把 row decoder 分成两部分：

- 普通 decoder：处理 `C/D-group`
- 小型 bitwise decoder：处理 `B-group`

执行时，控制器先让第一次激活进行到足够深的阶段，再快速发起第二次 `ACTIVATE`，让结果覆盖到目标行。这样两个激活过程可以部分重叠。论文基于 SPICE 估计，该优化可把 AAP 延迟从 80ns 压到 49ns。

#### 9. ISA 与系统栈配合：Ambit 并不只是一个器件技巧

论文没有把 Ambit 当成纯硬件黑盒，而是给出了一套系统路径。作者提出新的 bulk bitwise 指令形式：

- `bbop dst, src1, [src2], size`

当处理器发现：

- 地址按 DRAM row 对齐
- 操作大小是 DRAM row 大小的整数倍

就把请求交给内存控制器，由控制器翻译成一串 `AAP/AP/ACTIVATE/PRECHARGE` 操作；若不满足这些约束，则退回 CPU 自己执行。

在缓存一致性上，控制器执行 Ambit 前需要：

- flush 源行对应的脏 cache lines
- invalidate 目标行对应的 cache lines

论文认为这和 DMA 的一致性处理本质相似，可复用已有机制。

#### 10. 硬件代价与测试代价

Ambit 之所以被论文强调为“practical”，关键在于它只对 commodity DRAM 做了很少的改动：

- 在每个 subarray 中增加 4 个 designated rows
- 增加两行 DCC
- 让 row decoder 认识少量保留地址

作者估计，总体额外存储/面积开销约为每个 subarray 8 行，整体小于 DRAM 芯片面积的 `1%`。测试方面也只需要在原有 DRAM 测试流程上补充对 `B-group` 地址和 DCC 的验证，因此制造代价相对可控。

### 五、实验内容

#### 1. 微基准测试

论文比较了 Ambit 与以下平台在 bulk bitwise operations 上的 raw throughput：

- Intel Skylake CPU
- NVIDIA GTX 745 GPU
- HMC 2.0 logic layer
- 普通 DRAM 上的 Ambit
- 3D-stacked DRAM 上扩展的 Ambit-3D

测试方法是在大输入向量上反复执行多种 bulk bitwise operation，包括：

- `NOT`
- `AND / OR`
- `NAND / NOR`
- `XOR / XNOR`

#### 2. 吞吐与能耗结果

- Ambit 在七类 bulk bitwise operation 上平均性能提升约 32 倍，平均能耗降低约 35 倍。
- 相比 Skylake，Ambit 平均吞吐提升约 44.9 倍。
- 相比 GTX 745，平均吞吐提升约 32.0 倍。
- 相比 HMC 2.0 logic layer，平均吞吐提升约 2.4 倍。
- 若将 Ambit 集成进 3D-stacked DRAM，则可比 HMC logic layer 再提升约 9.7 倍。
- 在能耗方面，Ambit 相比传统 DDR3 通道搬运方案，对不同位运算的节能幅度约为 25.1 倍到 59.5 倍。

#### 3. 真实应用评估

论文基于 gem5 全系统仿真评估了 Ambit 在三类真实应用上的端到端收益：

- **Bitmap indices：** 平均性能提升约 6.0 倍。
- **BitWeaving：** 平均性能提升约 7.0 倍。
- **Bitvector-based set operations：** 相比已有 bitset 实现平均提升约 3.0 倍。

此外，论文还讨论了 Ambit 对以下应用的潜在价值：

- BitFunnel
- masked initialization
- encryption
- DNA sequence mapping

#### 4. 电路级可靠性验证

作者使用 SPICE 对 Ambit 做了电路级仿真：

- Ambit-NOT 与常规 DRAM 行为相近，可靠性较高。
- triple-row activation 在存在工艺波动时仍可工作。
- 在 Monte Carlo 仿真中，工艺波动为 ±5% 时未出现错误。
- 即使波动扩大到 ±10% 和 ±15%，错误率也分别仅为 0.29% 和 6.01%。

#### 5. 硬件成本评估

- Ambit 在每个 subarray 中增加少量 designated rows 和 DCC rows。
- 总面积开销小于 1% DRAM chip area。
- 不需要修改 DRAM command/address interface。
- 即使某些 Ambit 专用功能测试失败，芯片仍可能作为普通 DRAM 出货，从而降低良率风险。

### 六、存在的不足

**1. 对数据布局有较强依赖**

- 为了高效使用 RowClone-FPM，参与同一位运算的数据最好映射到同一个 subarray。
- 因此软件层或驱动层需要感知底层地址映射，并主动安排 bitvector 的放置方式。

**2. 操作粒度较粗，不适合细粒度小数据位运算**

- Ambit 的所有操作都是按 row 粒度执行的。
- 只有当源地址和目标地址按 DRAM row 对齐，且操作规模足够大时，才容易发挥收益。

**3. 与现有 ECC 和数据 scrambling 的兼容性不足**

- 传统 ECC 方案不能直接支持在内存内部原位进行 bitwise operation。
- 论文提到需要满足同态性的 ECC 机制，但实际低开销方案仍是开放问题。
- data scrambling 也会干扰 Ambit 对原始比特模式的直接解释。

**4. 可靠性仍然依赖器件工艺与测试机制**

- triple-row activation 的正确性依赖 bitline charge sharing 和 sense amplification 的稳定性。
- 虽然论文通过 SPICE 仿真证明其在一定工艺波动范围内可行，但它仍比普通单行激活更敏感。

**5. 适用范围偏专用，而非通用计算**

- Ambit 非常适合 bulk bitwise operation 这类行粒度、位向量式、数据并行的任务。
- 但它并不能直接替代通用处理器，也不适用于复杂控制流或细粒度标量计算。
